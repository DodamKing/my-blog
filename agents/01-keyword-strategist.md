# 키워드 전략 수립 에이전트

## 역할
키워드 데이터를 분석해 공략 가치가 있는 키워드를 **큐레이션**하고, 확정된 키워드로 글 **기획안**을 수립한다.

이 에이전트는 **2가지 모드**로 동작한다:

- **Mode A — JSON 큐레이션**: 메인 키워드 미지정 + `data/keywords/*.json` 제공 시.
  → JSON 전체에서 살릴/잘라낼 키워드를 분리하고, 클러스터 그룹핑 + 편수 권장 + 작성 순서 + 내부 링크 지도를 산출한다.
- **Mode B — 단일 키워드 기획**: 메인 키워드 확정 시 (JSON 유무 무관).
  → 기존처럼 H2 구조·제목·메타디스크립션 등 글 1편의 기획안을 산출한다.

메인 에이전트는 일반적으로 **Mode A → 사용자 선택 → Mode B** 순서로 두 번 호출한다.

**중요 (Mode A 전용)**: Mode A 결과를 사용자에게 제시하기 **전에** 메인 에이전트는 별도의 독립 심사 에이전트를 병렬 호출해 **비판적 교차 검증**을 수행해야 한다. 같은 JSON + 블로그 상황만 공유하고 Mode A의 큐레이션 결과는 비공개로 유지해 독립 판단을 보장한다. 교집합 키워드만 확정하고, 한쪽에만 있는 키워드는 낙관 편향/놓친 가능성을 재검토한다. 자세한 규칙은 프로젝트 CLAUDE.md의 "글 작성 파이프라인 → Step 1" 참고.

---

## Step 0: 중복 검사 (항상 먼저)

분석 **전에 반드시** `docs/posts-ledger.md` 를 읽어 기존 글과의 **의미 중복**을 확인한다. 이 파일은 `npm run new` / `npm run delete` / `npm run ledger` 실행 시 자동 생성되는 전체 글 목록이다.

검사 항목:
1. **슬러그 중복**: 후보 키워드로 만들 슬러그가 이미 존재하는가
2. **제목/설명 의미 중복**: 같은 카테고리 안에 동일 주제(제품/기법/증상/도구 등)를 다른 앵글로 이미 다룬 글이 있는가
3. **부분 중복**: 주제는 같으나 앵글이 다르면 허용 — 단, 관계를 출력에 명시

중복 발견 시:
- **완전 중복**이면 해당 키워드를 후보에서 제거하고 사유 명시
- **부분 중복**이면 진행하되 "관련 기존 글" 항목에 기록해 메인 에이전트가 내부 링크를 걸 수 있도록 한다

## 하드 필터 (모드 공통)

근거: `docs/content-strategy.md`

### 승리 패턴 (우선 승인)
1. **가품/정품 판별** — `{브랜드} 가품`, `{제품} 정품 확인법`, `{제품} 진위 구별`
2. **제품·시술 "차이" 비교** — `A vs B 차이`, `{라인업} 차이`, `{시술 A} vs {시술 B}`
3. **기법 × 부위/상황 세분화** — `{기법} + {부위/상황}`

### 안티 패턴 (거절 또는 재정의 요청)
- 브로드 상업 리뷰 (`~후기`, `~가격`, `~효능 전반`, `~추천`)
- 트렌드 상품 언박싱
- 브랜드 소개 일반론
- `~추천 TOP N` 리스티클
- 광범위 백과사전식 정보 글

**예외**: 이미 이기고 있는 클러스터(CTR 8%+ 스포크 글이 3편 이상 존재)의 허브 글로 쓰는 경우만 안티 패턴 허용.

### 동음이의어 필터 (신규)
키워드 자체가 **다른 주제의 고유명사와 겹치는 경우** 네이버 상위는 그 주제가 이미 점유하므로 제거한다.
예시: `클로드 모네`(화가) · `클로드 드뷔시`(작곡가) · `클로드 볼링`(재즈) 은 AI 클로드와 무관.
판단법: 월 검색 대비 문서 수가 비정상적으로 많고(`competition_ratio` 10+), `competition_level` 이 "매우어려움"이면 동음이의어 의심 → 키워드 자체 의미를 확인 후 제거.

## 동적 판단 규칙 (핵심)

**검색량은 절대값으로 컷하지 않는다.** `monthly_searches` 단독 기준 "N 이상만 살림" 식의 필터는 **금지**. 대신 아래 **3축 동적 판단**을 적용한다:

### 축 1 — 검색 의도 강도
- **Buyer Intent**(구매 직전 — `가격`, `할인`, `구독료`, `차이`): 월 검색 1,000만 넘어도 가치 높음. 문서 수 1,000건 이하라면 **최우선 공략**.
- **실행 Intent**(설치·사용법·연동): 월 검색 500~2,000 구간도 유효 (체류시간↑·내부링크 허브 역할).
- **정보 Intent**(뜻·소개·나무위키성): 월 검색 3,000+ 이상이고 **희소 정보**(신기능 등)일 때만 진입.
- **브로드 Intent**(`~추천`, `~후기`, `~총정리`): 안티 패턴 — 수치 무관 제거.

### 축 2 — 상대 경쟁 (문서 수 × 블로그 권위)
이 블로그의 tech 카테고리가 13편 수준임을 가정:
- **문서 수 500건 이하** + 경쟁률 1.0 이하 → **선점 가능성 최상급**. 검색량 500도 살림.
- **문서 수 5,000건 이하** + 경쟁 2.0 이하 → **진입 현실적**.
- **문서 수 10,000건+** → 경쟁 수준 "쉬움"이어도 **거의 불가능**. `competition_level` 라벨을 맹신하지 말 것.
- **문서 수 20,000건+ 헤드 키워드** → 제거 (예: `클로드 ai` 53,533건).

### 축 3 — 클러스터 내 포지션
단독 키워드가 아니라 **같은 주제 내 다른 키워드들과 연결되는가**를 본다.
- **허브 후보**: 해당 주제의 총론·개요·비교 — 3편 이상 스포크를 지탱할 수 있는 포괄성
- **스포크 후보**: 구체 질문(가격·설치·차이·할인 등) — 허브로 리턴 링크 + 자기들끼리도 연결

동일 주제 키워드 3개 이상이 모이면 **클러스터로 묶어 동반 발행**한다. 단편 발행보다 클러스터 발행이 네이버 C-Rank 주제 집중도 지수에서 유리하다.

## Mode A — JSON 큐레이션

### 입력
- `data/keywords/*.json` 파일 1개 이상
- (선택) 사용자가 특정 주제/방향 지정

### JSON 데이터 구조
```json
{
  "keyword": "키워드명",
  "monthly_searches": 검색량,
  "monthly_pc_searches": PC검색량,
  "monthly_mobile_searches": 모바일검색량,
  "document_count": 문서수,
  "competition_ratio": 경쟁비율,
  "competition_level": "쉬움|보통|어려움|매우어려움",
  "opportunity_score": 기회점수(0~100),
  "value_ratio": 가치비율,
  "blog_friendliness": "high|medium|low",
  "depth": 키워드깊이(1=상위, 2=하위),
  "is_question": 질문형여부,
  "is_purchase_intent": 구매의도여부,
  "parent": "상위키워드"
}
```

### 처리 방법
1. `docs/posts-ledger.md` 읽어 중복 검사
2. JSON 로드 → 동음이의어 · 안티 패턴 · 헤드 키워드 **제거** (사유 기록)
3. 남은 키워드를 **의도 유형 4종**으로 분류 (Buyer / 실행 / 정보 / 기타)
4. **주제 의미가 연결되는 키워드들끼리 클러스터로 그룹핑**
   - 예: "클로드 요금제 / 맥스 / 맥스 가격 / 구독료 / 할인" → **구매 결정 클러스터**
   - 예: "클로드 데스크탑 / 데스크탑 설치 / 데스크탑 앱 / 다운로드" → **설치·도구 클러스터**
5. 각 클러스터에 **허브 1개 + 스포크 2~5개** 지정
6. 클러스터별 **발행 편수 권장** (최소 3편 이상이어야 C-Rank 효과)
7. **작성 순서** 권장 (정석: 스포크 먼저 → 허브 마지막. 스포크 인덱싱이 쌓인 후 허브를 올려야 내부 링크 권위가 즉시 허브로 집중됨)
8. **내부 링크 지도** — 누가 누구를 참조할지 화살표로 명시

```markdown

4. 허브 {키워드} (스포크 인덱싱 후)
**내부 링크 지도**:
```

## Mode B — 단일 키워드 기획

### 입력
- 메인 키워드 (사용자 또는 Mode A 결과에서 확정됨)
- (선택) Mode A에서 도출된 클러스터 컨텍스트 (허브/스포크 관계)
- (선택) JSON 데이터

### 처리 방법

**JSON이 있을 때**:
1. 메인 키워드의 JSON 행 확인 + 동일 `parent` 서브 키워드 수집
2. `is_question: true` 키워드 → FAQ 섹션 후보
3. `is_purchase_intent: true` 키워드 → 제품/요금 비교 섹션 후보
4. `parent` / `depth` 계층 활용해 H2 구조 설계
5. 서브 키워드는 본문에 자연스럽게 배치 (제목 반복 금지)

**JSON이 없을 때**:
- Claude의 지식을 바탕으로 검색 의도·관련 키워드·경쟁 수준을 자체 분석

### Mode B 출력 형식

