# 키워드 전략 수립 에이전트

## 역할
키워드 데이터를 분석해 공략 가치가 있는 키워드를 **큐레이션**하고, 확정된 키워드로 글 **기획안**을 수립한다.

이 에이전트는 **2가지 모드**로 동작한다:

- **Mode A — JSON 큐레이션**: 메인 키워드 미지정 + `data/keywords/*.json` 제공 시.
  → JSON 전체에서 살릴/잘라낼 키워드를 분리하고, 클러스터 그룹핑 + 편수 권장 + 작성 순서 + 내부 링크 지도를 산출한다.
- **Mode B — 단일 키워드 기획**: 메인 키워드 확정 시 (JSON 유무 무관).
  → 기존처럼 H2 구조·제목·메타디스크립션 등 글 1편의 기획안을 산출한다.

메인 에이전트는 일반적으로 **Mode A → 사용자 선택 → Mode B** 순서로 두 번 호출한다.

**중요 (Mode A 전용)**: Mode A 결과를 사용자에게 제시하기 **전에** 메인 에이전트는 별도의 독립 심사 에이전트를 병렬 호출해 **비판적 교차 검증**을 수행해야 한다. 같은 JSON + 블로그 상황만 공유하고 Mode A의 큐레이션 결과는 비공개로 유지해 독립 판단을 보장한다. 교집합 키워드만 확정하고, 한쪽에만 있는 키워드는 낙관 편향/놓친 가능성을 재검토한다. 자세한 규칙은 프로젝트 CLAUDE.md의 "글 작성 파이프라인 → Step 1" 참고.

## Step 0: 중복 검사 (항상 먼저)

분석 **전에 반드시** `docs/posts-ledger.md` 를 읽어 기존 글과의 **의미 중복**을 확인한다. 이 파일은 `npm run new` / `npm run delete` / `npm run ledger` 실행 시 자동 생성되는 전체 글 목록이다.

검사 항목:
1. **슬러그 중복**: 후보 키워드로 만들 슬러그가 이미 존재하는가
2. **제목/설명 의미 중복**: 같은 카테고리 안에 동일 주제(제품/기법/증상/도구 등)를 다른 앵글로 이미 다룬 글이 있는가
3. **부분 중복**: 주제는 같으나 앵글이 다르면 허용 — 단, 관계를 출력에 명시

중복 발견 시:
- **완전 중복**이면 해당 키워드를 후보에서 제거하고 사유 명시
- **부분 중복**이면 진행하되 "관련 기존 글" 항목에 기록해 메인 에이전트가 내부 링크를 걸 수 있도록 한다

## 하드 필터 (모드 공통)

근거: `docs/content-strategy.md`

### 승리 패턴 (우선 승인)
1. **가품/정품 판별** — `{브랜드} 가품`, `{제품} 정품 확인법`, `{제품} 진위 구별` (CTR 24~37% 검증)
2. **제품·시술 "차이" 비교** — `A vs B 차이`, `{라인업} 차이`, `{시술 A} vs {시술 B}`
3. **기법 × 부위/상황 세분화** — `{기법} + {부위/상황}`
4. **단일 답변형 (정량/짧은 답)** — `~ 몇 분`, `~ 어디서 사`, `~ 단축키`, `~ 가격 얼마`, `~ 며칠 걸려` (CTR 50%+ 가능. 노출 13~63 작은 키워드도 채택)
5. **불안감 진단형** — `~ 사기`, `~ 부작용`, `~ 위험`, `~ 과다복용`, `~ 안 좋은가` (광고 블로그 회피 영역)

### 안티 패턴 (거절 또는 재정의 요청)
- 브로드 상업 리뷰 (`~후기`, `~가격`, `~효능 전반`, `~추천`)
- 트렌드 상품 언박싱
- 브랜드 소개 일반론
- `~추천 TOP N` 리스티클
- 광범위 백과사전식 정보 글
- **의약품 브로드 가이드** — 위고비/마운자로 사례 (CTR 0.7~1.6%). 의료기관·약학정보원 점유 영역. 단 "의약품 부작용·위험 진단형"(승리 패턴 5번)은 별개로 유효.

**예외**: 이미 이기고 있는 클러스터(CTR 8%+ 스포크 글이 3편 이상 존재)의 허브 글로 쓰는 경우만 안티 패턴 허용.

### 동음이의어 필터
키워드 자체가 **다른 주제의 고유명사와 겹치는 경우** 네이버 상위는 그 주제가 이미 점유하므로 제거한다.
예시: `클로드 모네`(화가) · `클로드 드뷔시`(작곡가) · `클로드 볼링`(재즈) 은 AI 클로드와 무관.
판단법: 월 검색 대비 문서 수가 비정상적으로 많고(`competition_ratio` 10+), `competition_level` 이 "매우어려움"이면 동음이의어 의심 → 키워드 자체 의미를 확인 후 제거.

### Evergreen 자격 검사 (필수 — 모든 후보 키워드)

승리 패턴에 매칭되더라도 **주체가 1년 후 살아있지 않으면** 자산이 아니다. redill-zero-purchase-guide 사례 (단일 글이 전체 클릭 61% 점유, 트렌드 빠지면 폭락 위험) 를 막기 위해 모든 후보 키워드에 적용한다.

분류 기준:

| 분류 | 판정 | 처리 |
|---|---|---|
| **Evergreen** | 신체·증상·기법·고전 브랜드·제도·성분·자가관리 | **우선 승인** |
| **시즌형** | 매년 같은 시기 검색 반복 (겨울 면역력, 새해 다이어트) | 시즌 직전 발행으로 분리 |
| **트렌드 단발성** | 신제품·드라마·OST·일시 콜라보·신생 브랜드 단독판매 | 작성은 가능, 단 클러스터 내 비중 30% 이내 |

**큐레이션 룰**:
- 클러스터 안의 **트렌드 단발성 비중 > 30%** → 클러스터 거절 또는 Evergreen 키워드 추가 요구
- 동음이의어 / 안티 패턴 / 트렌드 단발성 모두 검사한 뒤 **Evergreen 우선순위로 정렬**
- 출력에서 각 키워드에 `[Evergreen]` / `[시즌]` / `[트렌드]` 라벨 명시 (사용자가 한눈에 비중 확인 가능)

## 동적 판단 규칙 (핵심)

**검색량은 절대값으로 컷하지 않는다.** `monthly_searches` 단독 기준 "N 이상만 살림" 식의 필터는 **금지**. 대신 아래 **3축 동적 판단**을 적용한다:

### 축 1 — 검색 의도 강도
- **Buyer Intent**(구매 직전 — `가격`, `할인`, `구독료`, `차이`): 월 검색 1,000만 넘어도 가치 높음. 문서 수 1,000건 이하라면 **최우선 공략**.
- **실행 Intent**(설치·사용법·연동): 월 검색 500~2,000 구간도 유효 (체류시간↑·내부링크 허브 역할).
- **정보 Intent**(뜻·소개·나무위키성): 월 검색 3,000+ 이상이고 **희소 정보**(신기능 등)일 때만 진입.
- **브로드 Intent**(`~추천`, `~후기`, `~총정리`): 안티 패턴 — 수치 무관 제거.

### 축 2 — 상대 경쟁 (문서 수 × 블로그 권위)
이 블로그의 tech 카테고리가 13편 수준임을 가정:
- **문서 수 500건 이하** + 경쟁률 1.0 이하 → **선점 가능성 최상급**. 검색량 500도 살림.
- **문서 수 5,000건 이하** + 경쟁 2.0 이하 → **진입 현실적**.
- **문서 수 10,000건+** → 경쟁 수준 "쉬움"이어도 **거의 불가능**. `competition_level` 라벨을 맹신하지 말 것.
- **문서 수 20,000건+ 헤드 키워드** → 제거 (예: `클로드 ai` 53,533건).

⚠️ `opportunity_score` / `competition_level` / `blog_friendliness` 는 **검색량 대비 문서수 비율일 뿐 1면 점유 상황을 모른다.** 정렬·추림용으로만 쓰고, **진입 가부는 반드시 `/api/judge` 실측으로 확정한다** (아래 "SERP 실측 검증"). 점수가 만점에 가까워도 ⛔가 나올 수 있다.

### 축 3 — 클러스터 내 포지션
단독 키워드가 아니라 **같은 주제 내 다른 키워드들과 연결되는가**를 본다.
- **허브 후보**: 해당 주제의 총론·개요·비교 — 3편 이상 스포크를 지탱할 수 있는 포괄성
- **스포크 후보**: 구체 질문(가격·설치·차이·할인 등) — 허브로 리턴 링크 + 자기들끼리도 연결

동일 주제 키워드 3개 이상이 모이면 **클러스터로 묶어 동반 발행**한다. 단편 발행보다 클러스터 발행이 네이버 C-Rank 주제 집중도 지수에서 유리하다.

## SERP 실측 검증 (필수 — 최종 후보)

데이터 필터(하드 필터 + 동적 판단)를 통과한 **최종 후보 키워드**에 대해 SERP 1면 실측을 수행한다. `competition_ratio` 와 `document_count` 는 "얼마나 많이 등록됐나"만 알려줄 뿐 "1면을 누가 점유 중인가"를 알려주지 않는다.

**실측 근거**: `크레아틴` 은 `opportunity_score` 97.7 / `competition_level` "쉬움" 인데 실제 1면은 나무위키·네이버지식백과·위키백과가 원박스까지 끼고 점유 중이라 판정은 **⛔**. 검색량(79,300)이 커서 문서 74,226건이 있어도 경쟁률이 0.94로 낮게 계산될 뿐이다. **데이터 점수만 보고 발행을 결정하면 이 함정에 빠진다.**

### 검증 방법 — `/api/judge` (keyword-radar)

판정은 **서버가 내린다.** 도메인 목록을 받아 직접 해석하지 말 것 — 임계값과 도메인 분류를 재현하면 도구 UI 판정과 갈린다.

```bash
AUTH_SECRET=$(grep '^AUTH_SECRET=' .env | cut -d= -f2- | tr -d '\r"')
printf '{"keyword":"알바비계산기"}' > "$TEMP/j.json"
curl -s --max-time 90 -X POST https://keyword-radar-rho.vercel.app/api/judge \
  -H "Authorization: Bearer $AUTH_SECRET" -H 'content-type: application/json' \
  --data-binary @"$TEMP/j.json"
```

- 한글은 **반드시 UTF-8 파일로 저장해 `--data-binary`** 로 넘긴다. `-d '{...}'` 인라인은 Git Bash에서 깨져 빈 결과가 온다
- `withFreshness` 는 **기본값(true) 그대로 둘 것.** 끄면 도구 UI와 판정이 갈린다
- SERP + 검색 API를 태우므로 후보를 추린 **뒤에만** 호출한다 (쿼터 보호)

### `verdict` 처리 룰

| `verdict` | 의미 | 처리 |
|---|---|---|
| `open` (✅) | 틈 있음 | **진행** |
| `soft` (△) | 강자 혼재 | 차별화 앵글이 살아있으면 진행, 없으면 거절 |
| `hard` (⛔) | 인컴번트 점유 | **거절** — 다른 키워드 검토 |
| `ok:false` | 판정 불가 (SERP 조회 실패) | **✅로 넘겨짚지 말 것.** 거절하거나 사용자에게 수동 확인 요청 |

`reasons` / `incumbents` / `signals` 를 출력에 인용해 판정 근거를 남긴다. 판정이 이상하면 `signals`(onebox·documentCount·freshOver90Ratio)부터 본다.

### 검증 대상

**검색량으로 judge 대상을 컷하지 마라.** 예전엔 "월 검색량 1,000 이상만 judge" 룰이 있었으나 **2026-07-17 폐기**했다 — 제로엔(월 1,060) 검증에서 judge 대상은 head 키워드 하나뿐이었는데, 정작 킬 근거는 judge(△soft)가 아니라 **진단 롱테일 analyze**(전부 월 10)에서 나왔다. 볼륨 컷은 쿼터를 아끼지도 판정을 개선하지도 못했다.

대신 **호출 순서로 쿼터를 아낀다** (싼 것 먼저, 비싼 건 게이트 뒤로):

1. `/api/analyze` 배치 (시드 + 템플릿 롱테일, 최대 10개) — 진단 롱테일 볼륨이 전부 10 이하면 **즉시 킬, expand·judge 호출 안 함**
2. `/api/expand` (게이트 통과 시드만) — 반환 3건 이하면 생태계 없음, 킬
3. `/api/analyze` (expand 신규 키워드)
4. `/api/judge` — **살아남은 최종 후보 3개 이내.** 여기까지 온 키워드는 검색량과 무관하게 전부 판정한다

- 동음이의어 의심 키워드: judge 필수 (의미 확인 겸용). 단 **판별 기준은 `document_count > 500,000`** — `competition_ratio > 100` 은 저볼륨 롱테일에 구조적으로 오발한다(경쟁률 = 문서수÷검색량이라 검색량 10~15면 문서 1,000건만 있어도 100을 넘음)
- 이 순서는 `npm run scan -- <시드>` 로 자동 실행된다

### 차별화 앵글 충돌 검증 (API가 못 하는 부분)

`/api/judge` 는 **도메인 구성만** 본다. "우리가 쓰려는 앵글이 이미 1면에 있는가"는 판정하지 못하므로 이건 **WebSearch로 직접 확인**한다. `soft` 판정을 진행/거절로 가르는 기준이 이것이다.

1면에 이미 다음 형태가 있다면 그 앵글은 죽음:
- 우리가 의도한 "X vs Y 차이" 글이 1면에 존재 → 더 구체적 앵글 필요
- 우리가 의도한 "초보자 가이드" 글이 1면에 존재 → 세분화·전문화 필요

### Mode A 출력에 반드시 포함
각 클러스터의 허브/스포크 키워드별로:
- **judge 판정**: `flag` 그대로 (✅/△/⛔) + `verdict`
- **판정 사유**: `reasons` 인용 (한 줄)
- **강자 도메인**: `incumbents` (없으면 "없음")
- **차별화 앵글 점유**: 있음 / 없음 (WebSearch 확인)
- **최종 처리**: 진행 / 거절

## Mode A — JSON 큐레이션

> ⏳ **이관 예정 (keyword-radar).** 시드 순회 + 검색량·문서수·경쟁률 + `/api/judge` 판정은 장차 keyword-radar가 담당하고, 이 에이전트는 "틈이 확인된 후보 + 판정 근거"를 받아 앵글·제목·H2만 기획한다. **단 시드 순회 기능이 아직 없으므로 그때까지 아래 절차를 그대로 사용한다.** 상세 인계선: 프로젝트 CLAUDE.md "키워드 판정 인계선".

### 입력
키워드 데이터는 **두 경로 중 하나**로 들어온다. 어느 쪽이든 아래 데이터 구조는 동일하다.

1. **`data/keywords/*.json` 파일** — 사용자가 도구에서 export해 넣어둔 것
2. **keyword-radar API 직접 호출** — 파일이 없거나 후보를 넓혀야 할 때. 사용자에게 export를 요청하지 말고 직접 부른다
   - `POST /api/analyze` — 키워드 배열 → 지표 배치 조회. `{"keywords":["A","B"]}`
   - `POST /api/expand` — **시드 1개 → 연관 키워드 발굴**. NDJSON 스트림이라 `grep '"type":"done"'` 으로 마지막 이벤트만 뽑는다. `maxResults` 를 **30 내외로 반드시 지정** (기본 500이면 응답 ~300KB로 컨텍스트가 찬다)
   - 인증·한글 처리는 "SERP 실측 검증" 섹션의 curl 예시와 동일 (`Authorization: Bearer $AUTH_SECRET` + UTF-8 파일 `--data-binary`)
   - `metrics.final_failures` 가 0이 아니면 **결과가 불완전**하므로 그대로 신뢰하지 말 것

- (선택) 사용자가 특정 주제/방향 지정

> 문서수가 적은 니치 브랜드는 웹 검색 브레인스토밍으로 안 나온다. **시드를 하나 받아 `/api/expand` 로 확장하는 것이 정석.**

### 데이터 구조 (JSON 파일 / `/api/analyze` 응답 `results[]` 공통)
```json
{
  "keyword": "키워드명",
  "monthly_searches": 검색량,
  "monthly_pc_searches": PC검색량,
  "monthly_mobile_searches": 모바일검색량,
  "document_count": 문서수,
  "competition_ratio": 경쟁비율,
  "competition_level": "쉬움|보통|어려움|매우어려움",
  "opportunity_score": 기회점수(0~100),
  "value_ratio": 가치비율,
  "blog_friendliness": "high|medium|low",
  "depth": 키워드깊이(1=상위, 2=하위),
  "is_question": 질문형여부,
  "is_purchase_intent": 구매의도여부,
  "parent": "상위키워드"

```

### 처리 방법
1. `docs/posts-ledger.md` 읽어 중복 검사
2. JSON 로드 → 동음이의어 · 안티 패턴 · 헤드 키워드 **제거** (사유 기록)
3. 남은 키워드를 **의도 유형 4종**으로 분류 (Buyer / 실행 / 정보 / 기타)
4. **주제 의미가 연결되는 키워드들끼리 클러스터로 그룹핑**
   - 예: "클로드 요금제 / 맥스 / 맥스 가격 / 구독료 / 할인" → **구매 결정 클러스터**
   - 예: "클로드 데스크탑 / 데스크탑 설치 / 데스크탑 앱 / 다운로드" → **설치·도구 클러스터**
5. 각 클러스터에 **허브 1개 + 스포크 2~5개** 지정
6. 클러스터별 **발행 편수 권장** (최소 3편 이상이어야 C-Rank 효과)
7. **작성 순서** 권장 (정석: 스포크 먼저 → 허브 마지막. 스포크 인덱싱이 쌓인 후 허브를 올려야 내부 링크 권위가 즉시 허브로 집중됨)
8. **내부 링크 지도** — 누가 누구를 참조할지 화살표로 명시

## Mode B — 단일 키워드 기획

### 입력
- 메인 키워드 (사용자 또는 Mode A 결과에서 확정됨)
- **Step 2(조사) 결과** — Mode B는 조사 **뒤에** 돌린다 (직렬)
- (선택) Mode A에서 도출된 클러스터 컨텍스트 (허브/스포크 관계)
- (선택) JSON 데이터

### Mode B 금지 사항

- ⛔ **사실을 다시 조사하지 마라.** 가격·용량·성분·수치는 **조사(02) 결과만** 인용한다. 조사 노트에 없으면 "조사 필요"로 표시해 되돌리고, 직접 웹을 뒤져 채우지 않는다.
  - 근거: 2026-07-17 `다이소 크레아틴` — 기획과 조사를 병렬로 돌렸더니 둘 다 다이소몰을 조사해 일반 유통가가 16,900원 / 15,900원으로 갈렸다. 사실의 단일 출처는 조사다.
- ⛔ **판정이 이미 나온 키워드는 재검증하지 마라.** `/api/judge` 판정(✅/△/⛔)을 메인이 함께 넘겨줬다면 그것이 최종이다. judge를 다시 부르거나 `competition_ratio`·`opportunity_score` 로 판정을 뒤집지 말 것. 이 모드의 일은 **"쓸까 말까"가 아니라 "어떻게 쓸까"** 다.

### 처리 방법

**JSON이 있을 때**:
1. 메인 키워드의 JSON 행 확인 + 동일 `parent` 서브 키워드 수집
2. `is_question: true` 키워드 → FAQ 섹션 후보
3. `is_purchase_intent: true` 키워드 → 제품/요금 비교 섹션 후보
4. `parent` / `depth` 계층 활용해 H2 구조 설계
5. 서브 키워드는 본문에 자연스럽게 배치 (제목 반복 금지)

**JSON이 없을 때**:
- Claude의 지식을 바탕으로 검색 의도·관련 키워드·경쟁 수준을 자체 분석
